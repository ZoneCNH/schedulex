# 架构设计

> schedulex 核心设计决策记录（ADR 格式）

---

## ADR-1：接口驱动设计

**状态：** 已采纳

**背景：**
调度器需要在不同环境中灵活替换核心组件——测试需要确定性时钟，生产需要墙钟；触发策略需要支持间隔、定时、Cron 等多种形式；任务实现千差万别。

**决策：**
定义 6 个核心接口：`Clock`、`Job`、`Trigger`、`EventSink`、`Locker`、`Lease`。

**理由：**
- 接口隔离：每个接口职责单一，变更互不影响
- 可测试性：`StaticClock` + `Once` 触发器可在无墙钟的情况下验证调度逻辑
- 可扩展性：新增触发器类型（如 `Cron`）无需修改调度器核心
- 依赖倒置：调度器依赖抽象而非具体实现

**后果：**
- 用户需理解接口契约才能正确实现扩展
- 接口数量较多，初次接触有学习成本

## ADR-2：Functional Options 模式

**状态：** 已采纳

**背景：**
调度器和任务均有多个可选配置项，需要一种向后兼容、可渐进扩展的配置方式。

**决策：**
使用 `Option` 和 `JobOption` 函数类型（Functional Options 模式）进行配置。

**理由：**
- 向后兼容：新增配置项不破坏现有 API
- 类型安全：编译期检查配置合法性
- 自文档化：`WithClock(clock)` 比 `Config{Clock: clock}` 更清晰
- 验证前置：选项函数在构造时验证，非法配置提前报错

**替代方案：**
- 配置结构体：字段暴露过多，无法在构造时验证
- Builder 模式：代码冗长，Go 社区不常用

**后果：**
- 选项函数数量随功能增长而增加
- 需要文档明确各选项的默认值和约束

## ADR-3：策略模式（Clock/Trigger/Job 可替换性）

**状态：** 已采纳

**背景：**
调度器的核心行为由三个维度决定：时间源（何时）、触发策略（多久一次）、任务逻辑（做什么）。这三个维度需要独立变化。

**决策：**
将 Clock、Trigger、Job 设计为独立可替换的接口，通过依赖注入组合。

**理由：**
- 正交性：修改触发策略不影响时钟或任务逻辑
- 测试友好：`StaticClock` + `Every(5s)` + mock Job 可精确验证调度行为
- 生产灵活：同一任务可在不同环境使用不同时钟源

**后果：**
- 接口组合的正确性由用户保证（如 Clock 和 Trigger 时区不一致）

## ADR-4：Observer 模式（EventSink 事件通知）

**状态：** 已采纳

**背景：**
调度器的生命周期事件（调度、启动、成功、失败、关闭等）需要对外暴露，用于监控、日志和发布证据采集。

**决策：**
使用 `EventSink` 接口实现观察者模式，支持调度器级别和任务级别的事件接收器。

**理由：**
- 解耦：事件生产者和消费者分离
- 多消费者：调度器级 sink + 任务级 sink 可同时生效
- 函数适配器：`EventSinkFunc` 允许用闭包快速实现
- 事件丰富：`Event` 结构体包含调度时间、启动时间、耗时、延迟、重试次数等完整上下文

**后果：**
- 事件回调在调度循环内同步执行，慢回调可能影响调度时序
- 用户需确保 EventSink 实现的幂等性和性能

## ADR-5：Semaphore 并发控制（channel 而非 Mutex）

**状态：** 已采纳

**背景：**
调度器需要限制并发执行的任务数量（`MaxConcurrent`），防止资源耗尽。

**决策：**
使用 buffered channel 作为信号量（semaphore）控制并发，而非 Mutex。

**理由：**
- 天然阻塞：`sem <- struct{}{}` 在容量满时自动阻塞，无需额外条件变量
- 取消感知：`select` 可同时监听 `sem` 和 `ctx.Done()`，支持优雅关闭
- 简洁性：一个 channel 替代 mutex + counter + condition variable
- 惯用 Go：channel-based 并发是 Go 社区推荐模式

**替代方案：**
- `sync.Mutex` + 计数器：需额外处理等待和取消逻辑
- `golang.org/x/sync/semaphore`：引入外部依赖，功能超出需求

**后果：**
- `MaxConcurrent` 在构造时确定，运行时不可变更

## ADR-6：Misfire 策略

**状态：** 已采纳

**背景：**
当调度器因休眠、负载过高或时钟偏差错过任务触发时间时，需要明确的行为定义。

**决策：**
提供三种 Misfire 策略：`Skip`、`RunOnce`、`CatchUp`。

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| `MisfireSkip` | 跳过所有错过时刻 | 非关键任务，错过即过 |
| `MisfireRunOnce` | 仅执行最后一个错过时刻 | 报表生成等只需最新数据的任务 |
| `MisfireCatchUp` | 依次执行所有错过时刻（上限 128） | 金融对账等不能丢失执行的任务 |

**理由：**
- 不同业务场景对失火的容忍度不同
- `CatchUp` 设置 128 次上限防止雪崩
- `misfireGrace`（100ms）避免因微小时钟偏差误判失火

**后果：**
- `CatchUp` 策略在长时间失火时可能产生大量执行
- Misfire 检测依赖时钟精度

## ADR-7：确定性 Jitter

**状态：** 已采纳

**背景：**
大量任务在同一时刻触发会导致"惊群效应"（thundering herd），需要引入抖动分散负载。传统随机抖动不可重现，难以调试和生成发布证据。

**决策：**
使用 FNV-64a 哈希基于 `jobID`、`Seed` 和运行序号生成确定性抖动。

**理由：**
- 确定性：相同输入产生相同输出，测试和发布证据可重现
- 无随机依赖：不依赖 `math/rand`，避免种子管理复杂性
- 分散性：FNV 哈希分布均匀，有效分散触发时刻
- 可调试：给定 `(jobID, Seed, run)` 可精确计算抖动值

**替代方案：**
- `math/rand` + 固定种子：Go 1.20+ 全局 rand 行为变更，兼容性风险
- 真随机抖动：不可重现，无法生成 golden file

**后果：**
- 抖动范围在任务注册时确定，运行时不可动态调整

## ADR-8：合约测试（JSON Schema + Golden Files）

**状态：** 已采纳

**背景：**
调度器的 Snapshot 和 Event 输出是发布证据的关键组成部分，需要确保跨版本的格式兼容性。

**决策：**
使用 JSON Schema 验证 Snapshot/Event 结构，Golden Files 确保确定性行为的跨版本一致性。

**理由：**
- JSON Schema：声明式定义输出格式，自动验证字段类型和必填项
- Golden Files：已知输入 → 已知输出的快照，回归测试发现意外变更
- 发布证据：Schema + Golden File 构成可审计的合约证据
- 工具链友好：JSON Schema 可用于下游消费者的代码生成

**后果：**
- Schema 变更需要版本化管理
- Golden File 更新需人工审核确认为有意变更
